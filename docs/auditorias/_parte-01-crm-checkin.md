# Parte 1 — CRM + Checkin

## Resumo

Escopo: `src/features/crm/` (52 arquivos, ~10.5k linhas) e `src/features/checkin/` (31 arquivos, ~7.6k linhas). O código funciona e é bem coberto por testes na camada de `lib/` (funções puras), mas os containers e sections são grandes demais (vários arquivos com 800-1500 linhas), misturam UI com regra de negócio e persistência, e há bastante repetição estrutural entre hooks de dados. Achados por severidade: **Alto** 6 (2 arquivos mortos de ~1000 linhas somadas, `any` disperso em formulários, hook `useCheckinPage` com 12+ responsabilidades), **Médio** ~14 (duplicação de padrão de hooks CRUD, componentes gigantes, erro engolido silenciosamente), **Baixo** ~10 (nomenclatura, acessibilidade pontual, ausência de testes em componentes visuais).

## 1. DRY

- `src/features/crm/hooks/useClientes.ts`, `useOportunidades.ts`, `useAgendamentos.ts`, `useAtendimentos.ts`, `useVendedorPerfil.ts`, `useMeuScore.ts`, `useVeiculosEstoque.ts`, `useCadenciaAgenda.ts` — todos repetem o mesmo esqueleto: `useState` para dado/loading/error, `fetchX` com `supabase.from(...).select(...).eq('seller_user_id', ...)`, `useEffect(() => { fetchX() }, [fetchX])`, retorno `{ data, loading, error, refetch, ... }`. Sugestão: extrair um `useSupabaseResource<T>(table, query, deps)` genérico ou um `createSupabaseListHook` factory para eliminar a repetição em ~8 hooks.
- `src/features/checkin/sections/CheckinHeader.tsx:35-43` e `src/features/checkin/sections/CheckinAdjustmentTab.tsx:42-50` — array `ADJUSTMENT_REASONS` duplicado literalmente (mesmos 7 valores). Sugestão: mover para `src/features/checkin/lib/` como constante compartilhada (mas note que `CheckinAdjustmentTab.tsx` é código morto — ver seção 2; melhor remover o arquivo morto do que unificar).
- `src/features/crm/CarteiraClientes.container.tsx:87-89` define `FiltroPrioridade`, `FiltroSituacao`, `FiltroPeriodo` localmente; a lógica de derivação de prioridade/situação replica conceitos já presentes em `src/features/crm/lib/mentorComercial.ts` (`Prioridade`, `derivarSituacao`). Não é duplicação literal, mas há forte acoplamento implícito entre os dois — vale documentar ou mover os tipos de filtro para o mesmo módulo de `mentorComercial.ts`.
- Lógica de "buscar liberação de fechamento por vendedor/data" (`supabase.from('fechamento_liberacoes')...maybeSingle()`) aparece quase idêntica em `src/features/checkin/hooks/useCheckinPage.ts:313-329` e `src/features/checkin/sections/CheckinHeader.tsx:201-220`. Sugestão: extrair `useLiberacaoStatus(vendedorId, data)` compartilhado.

## 4. Componentes

- `src/features/crm/CarteiraClientes.container.tsx` — 1509 linhas. Um único arquivo mistura: container principal (`CarteiraClientes`), painel de filtros (`PainelFiltros`), chips de filtro, painel de ficha do cliente (`FluxoClientePanel`, ~250 linhas), formulário de edição (`FormularioEdicaoFicha`), modal de próxima oportunidade (`ProximaOportunidadeModal`) e várias funções utilitárias de classificação (`calcularQualidadeFicha`, `calcularUrgenciaFicha`). Sugestão: quebrar em `CarteiraClientes/` com arquivos separados por componente (`FluxoClientePanel.tsx`, `PainelFiltros.tsx`, `ProximaOportunidadeModal.tsx`, `FormularioEdicaoFicha.tsx`) e um `carteira.utils.ts` para as funções de classificação.

Sugestão: separar cada modal em componente próprio recebendo apenas os handlers necessários, e extrair a lógica de "nova atividade" (steps, busca de cliente por telefone) para um hook `useNovaAtividadeForm`.
- `src/features/checkin/sections/CheckinCrmSection.tsx` — 1376 linhas, um único componente cobre: tabela responsiva (mobile + desktop) de clientes, linha expansível com edição inline, modal de cadastro/edição completo (formulário com 10+ campos visuais + campos ocultos espelhados para testes legados), modal de coerência venda-sem-atendimento, e toda a lógica de submit/validação (`handleCadastrar`, ~190 linhas). Sugestão: separar `ClientesTable` (mobile/desktop), `ClienteFormModal` e extrair `handleCadastrar` para um hook/serviço testável isoladamente.
- `src/features/checkin/hooks/useCheckinPage.ts` — não é componente, mas é o "cérebro" que concentra 12+ responsabilidades num hook só: formulário, drafts numéricos, deadline/lock stage, liberação de gerente (fetch Supabase), WhatsApp para o gerente, cálculo de disciplina, cálculo de contadores CRM do dia, submit/validação em cascata (8 validações sequenciais em `submitCheckin`), salvar rascunho em localStorage e salvar ajuste técnico. Sugestão: quebrar em hooks menores — `useCheckinDeadline`, `useCheckinLiberacao`, `useCheckinDisciplina`, `useCheckinDraft` — compostos pelo hook principal.
- `src/features/checkin/sections/CheckinHeader.tsx` — 843 linhas, mistura header visual, modal de histórico (list view) e formulário de regularização/correção retroativa (form view) num único componente com estado local próprio (`formValues`, `selectedRow`, `liberacaoStatus`). Sugestão: extrair o modal de histórico para `CheckinHistoryModal.tsx` e o formulário de correção para `CheckinRegularizacaoForm.tsx`.

## 5. Estado

- `src/features/checkin/hooks/useCheckinPage.ts` retorna um objeto com 45+ propriedades (linhas 762-826) consumido via destructuring parcial em `CheckinForm.tsx` e `Checkin.container.tsx` — isso é efetivamente "prop drilling via contexto de retorno de hook": qualquer novo dado precisa ser adicionado nesse objeto gigante e propagado manualmente. Sugestão: agrupar por domínio (`{ form, deadline, disciplina, crm, draft }`) em vez de um único nível achatado.
- `src/features/checkin/sections/CheckinHeader.tsx:60-79` mantém `formValues` (11 campos) como estado local próprio para o formulário de regularização, duplicando a forma de `CheckinForm` (que já tem seu próprio `form` via `ctx`). Não há vazamento de estado entre os dois, mas a duplicação estrutural sugere que ambos deveriam usar o mesmo tipo/shape de formulário de fechamento.
- `src/features/crm/CarteiraClientes.container.tsx:930-956` — `FluxoClientePanel` recebe `cliente`, `oportunidade`, `progresso` via props (bom) mas internamente reconstrói um `form` local inicializado a partir de `cliente` (linhas 958-969) que fica dessincronizado se `cliente` mudar enquanto `editando` é `true` — não há `useEffect` para resync. Efeito prático: se o painel permanecer aberto e os dados do cliente mudarem via refetch, o formulário de edição pode mostrar dados antigos.
- Não há uso indevido de Context para dado não-global neste escopo (o projeto usa hooks + props, não Context API, para esses dados).

## 6. Hooks

- Nenhuma violação das regras dos hooks (chamadas condicionais/em loop) foi encontrada nos arquivos lidos.
- `src/features/crm/CentralExecucao.container.tsx:366-368` — `useEffect(() => { if (currentSlot) setExpandedStep(currentSlot.key) }, [currentSlot?.key])`. A dependência é `currentSlot?.key`, mas o efeito também lê `currentSlot` (só o objeto, não só a key) — funcionalmente correto pois só usa `.key`, mas o eslint `exhaustive-deps` tende a reclamar disso; deixar explícito com um comentário ou usar `currentSlot` inteiro no array evitaria ambiguidade.
- `src/features/checkin/hooks/useCheckinPage.ts:184-190` e `:208-219` — dois `useEffect` distintos que reagem a mudanças de `referenceDate`/`selectedDate` e se sobrepõem parcialmente (um seta `customReferenceDate`, outro decide `historicalCheckin`). Não é bug, mas a interdependência entre os dois efeitos (um preenche estado que o outro consome) é difícil de rastrear — candidato a consolidar em uma máquina de estados (`useReducer`) para o fluxo de seleção de data/histórico.
- `src/features/checkin/hooks/useCheckinPage.ts` como um todo é um caso claro de "hook que deveria ser dividido" (ver seção 4) — a quantidade de `useMemo`/`useEffect` interdependentes (12 useEffect/useMemo só nesse arquivo) o torna difícil de testar e de garantir dependências corretas.
- Padrão consistente de "fetch on mount" (`useEffect(() => { fetchX() }, [fetchX])`) está correto em todos os hooks CRM verificados — sem deps faltando.

Mesmo padrão de acoplamento lógica+UI do item acima.
- `src/features/checkin/hooks/useCheckinPage.ts:339-381` — `avisarGerenteWhatsapp` mistura chamada RPC ao Supabase, busca de gerente da loja, montagem de mensagem de texto e abertura de `window.open` — é regra de negócio + integração externa dentro do que deveria ser um hook de estado de página. Já está fora do componente visual (é hook), o que é parcialmente positivo, mas ainda mistura 3 responsabilidades (dados, formatação de mensagem, side-effect de navegador).

## 8. Erros

- `src/features/checkin/sections/CheckinCrmSection.tsx:534` — `catch (e) { toast.error('Erro ao cadastrar cliente.') }` captura a exceção em `e` mas nunca loga/usa a variável — se o erro for algo além do esperado (ex. erro de rede vs. erro de validação), a causa raiz se perde. Sugestão: `console.error('Erro ao cadastrar cliente', e)` antes do toast, ou reportar a um serviço de monitoramento.
- `src/features/checkin/hooks/useCrmDerivedTotals.ts:89-92` — quando qualquer uma das 4 queries paralelas falha (`clientesResult.error || opResult.error || ...`), o hook silenciosamente retorna `EMPTY` (zeros) sem logar o erro nem notificar o usuário. Isso pode fazer o formulário de checkin parecer "sem dados do CRM" quando na verdade houve falha de rede/permissão — o vendedor não tem como saber. Sugestão: pelo menos `console.error` e, se recorrente, expor um estado de erro consumível pela UI.
- `src/features/checkin/sections/CheckinHeader.tsx:337-340` — bloco `catch (err) { toast.error(...); console.error(err) }` é o único ponto do escopo que faz tratamento completo (toast + log). Vale usar como padrão de referência para os demais `catch` do projeto.
- Padrão geral positivo: a maioria das funções assíncronas em hooks (`useClientes`, `useOportunidades`, `useAgendamentos`) retornam `{ error: string | null }` em vez de lançar exceção, e os componentes consistentemente checam `if (error) { toast.error(error); return }` — isso é uma boa prática já estabelecida, sem achados negativos aqui.

## 9. Performance

- `src/features/crm/CarteiraClientes.container.tsx` — o componente principal (`CarteiraClientes`) não é memoizado e re-renderiza a lista inteira de clientes a cada mudança de estado local (busca, filtro, seleção de painel). Como o `map` de `listaBuscada` (linha 681) já reconstrói várias classificações (`calcularScoreCliente`, `derivarSituacao`, `explicacaoCliente`) por item a cada render, listas grandes de carteira poderiam se beneficiar de `React.memo` no card de cliente extraído como componente próprio, ou `useMemo` por item. Hoje esse card é JSX inline dentro do `.map`, o que impede memoização granular — reforça a sugestão da seção 4 de extrair `ClienteCard` como componente isolado.
- `src/features/checkin/sections/CheckinCrmSection.tsx` — nenhuma paginação/virtualização na tabela de clientes (`clientesList.map`, linha 707); para o volume atual (poucos clientes por fechamento diário) não é crítico, mas se a lista crescer (ex. carteira grande importada) pode degradar.
- Nenhuma imagem não otimizada encontrada no escopo (não há uso de `<img>` em `crm/` ou `checkin/`).
- Não há uso óbvio de função recriada a cada render causando problema de performance real (a maioria dos handlers já é definida no corpo do componente sem `useCallback`, mas como os componentes filhos não são memoizados, isso é consistente e não é uma causa isolada de re-render extra hoje).

## 10. Organização

- Estrutura de pastas é consistente entre as duas features (`hooks/`, `lib/`, `sections|components/`, containers na raiz) — sem inconsistência estrutural relevante.
- Import cruzado entre features: `src/features/crm/CentralExecucao.container.tsx:61-62` importa de `@/features/checkin/lib/clientes-list-from-crm` e `@/features/checkin/hooks/useCheckinPage` (para `timestampMatchesDateOnly` e `getSPHoursMinutes`). Isso cria acoplamento checkin→crm e crm→checkin (via essas duas funções utilitárias), o que é um cheiro de dependência cruzada entre features que deveriam ser independentes. Sugestão: mover `timestampMatchesDateOnly` e `getSPHoursMinutes` para um módulo compartilhado neutro (ex. `src/lib/datetime.ts`), já que são utilitários de data/hora sem relação direta com "checkin" como domínio.
- Nenhuma dependência circular real detectada (a direção é sempre checkin importando de crm, mais o caso pontual acima na direção contrária).
- `src/features/checkin/sections/CheckinAdjustmentTab.tsx` e `CheckinSidebar.tsx` sendo código morto (seção 2) também é, por si, um problema de organização: aumentam a superfície do diretório `sections/` sem propósito ativo.

## 11. Acessibilidade

- `src/features/crm/CarteiraClientes.container.tsx:177` — `<div className="flex-1 bg-black/30" onClick={onFechar} />` é um overlay de modal sem `role`/`aria-label` e sem suporte a teclado (Esc não fecha, não há foco preso no painel). Mesma observação para o overlay em `CarteiraClientes.container.tsx:1434-1435`, embora este já tenha `role="dialog" aria-modal="true"`.
- `src/features/checkin/sections/CheckinCrmSection.tsx:306` — `handleDelete` usa `window.confirm(...)` nativo para confirmação de exclusão; funciona mas não segue o padrão de modal acessível/estilizado usado no resto do app (`Modal` de `@/components/organisms/Modal`), gerando inconsistência de experiência (o `window.confirm` bloqueia a thread e tem estilo de SO, diferente dos modais React do resto da tela).
- Pontos positivos a destacar: a maioria dos botões de ícone no escopo já usa `aria-label` corretamente (ex. `CarteiraClientes.container.tsx:1011,1144,1162`; `CheckinCrmSection.tsx:617,620`), e não há `<img>` sem `alt` porque não há `<img>` no escopo.
- `src/features/checkin/sections/CheckinCrmSection.tsx:711-715` — a linha da tabela (`<tr onClick={...}>`) que expande detalhes ao clicar não tem `role="button"`/`tabIndex`/handler de teclado — usuários de teclado não conseguem expandir a linha (só o clique do mouse funciona). Sugestão: adicionar `tabIndex={0}`, `role="button"` e `onKeyDown` para Enter/Espaço, ou mover o toggle para um botão explícito dentro da linha.
