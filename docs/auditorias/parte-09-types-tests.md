# Parte 9 — Types (não gerados) + Tests

## Resumo

`src/types/` manualmente escrito (4 arquivos, ~800 linhas) está limpo: zero uso de `any`, convenção `interface` para entidades de domínio e `type` para unions consistente, sem tipos órfãos relevantes. O único achado real é duplicação estrutural entre `database.ts` e `database.generated.ts` (Supabase) que pode divergir silenciosamente do schema real. Em `src/test/` (38 arquivos, ~3945 linhas), a suíte de componentes (atoms/molecules/organisms) é de boa qualidade — testa classes CSS e comportamento real, não é "renderiza sem quebrar". Os testes de segurança/E2E (`security/`, `e2e/`, playwright na raiz) usam banco Supabase real sem mock, o que é positivo para RLS mas os torna dependentes de rede/seed. O problema maior está fora de `src/test/` mas é o achado mais crítico do ponto 12: hooks de autenticação/autorização (`useAuthRBAC`, `useAuthProfile`, `useAuthSession`, `useAuthActions`) e o serviço de check-in legado (`checkin-service.ts`) não têm nenhum teste unitário, apesar de concentrarem lógica de simulação de papel, cálculo de `effectiveRole`/`effectiveStoreId` e deduplicação de lançamento diário.

## Ponto 3 — TypeScript (src/types)

- `src/types/database.ts` e `src/types/database.generated.ts` — duplicação de tipos: `User`, `Store`, `DailyCheckin`, `Feedback`, `PDI`, `Training`, etc. são redeclarados manualmente em `database.ts` em vez de derivados via `Database['public']['Tables']['usuarios']['Row']` do arquivo gerado pelo Supabase. Risco: colunas renomeadas/adicionadas nas migrations não quebram o build (o tipo manual continua "válido") até o runtime falhar. Sugestão: ao menos os campos obrigatórios de entidades 1:1 com tabelas (`User`, `Store`, `DailyCheckin`, `Feedback`, `PDI`) deveriam usar `Pick`/`Omit` sobre `Tables<'usuarios'>` etc., mantendo apenas os tipos de UI/derivados (`RankingEntry`, `FunnelData`, `CheckinFormData`) como manuais.
- `src/types/database.ts:10-32` — múltiplos union types de enum (`UserRole`, `RoleCode`, `MxRoleCode`, `CheckinScope`, `CorrectionStatus`, etc.) sem nenhuma referência cruzada ao `Database['public']['Enums']` gerado. Se um enum Postgres ganhar um novo valor (ex.: nova `role`), o tipo manual não avisa em compile-time. Sugestão: gerar esses unions a partir de `Database['public']['Enums']['user_role']` quando o enum existir no banco.
- `src/types/database.ts:456` — `PDIReview` usa `[key: string]: unknown` como index signature aberta, o que anula o benefício de tipagem estrita da interface (qualquer propriedade passa a ser aceita). Sugestão: remover a index signature e listar os campos conhecidos explicitamente, ou usar `Record<string, unknown>` só no campo específico que realmente é dinâmico.
- `src/types/index.ts` (arquivo legado de mock/protótipo: `Task`, `Commission`, `Lead`, `Agency`, `InventoryItem`, `TeamMember`, `UIState`, `DailyLeadVolume`) — nenhum destes tipos é usado no restante de `src` (confirmado via grep — zero imports de `@/types/index` ou `from './index'` referenciando `Task`, `Commission`, `CommissionRule`, `Goal`, `Agency`, `Lead`, `LeadStage`, `InventoryItem`, `UIState`, `DailyLeadVolume` fora do próprio arquivo). Parece resíduo do protótipo Base44 anterior à migração para `database.ts`. Sugestão: remover o arquivo ou os tipos comprovadamente mortos, mantendo apenas o que for referenciado.
- `src/types/index.ts:45-51` — `export interface User { role: 'Owner' | 'Manager' | 'Seller' | 'RH' | 'Admin' }` colide conceitualmente com `User` de `database.ts` (mesmo nome, `role` com valores completamente diferentes: inglês genérico vs. enum PT-BR real do sistema). Como os dois arquivos são independentes (sem re-export cruzado), o TypeScript não acusa conflito, mas um import acidental de `@/types` em vez de `@/types/database` produziria um `User` com formato incompatível sem erro aparente até o uso em runtime. Sugestão: remover `User` de `index.ts` já que é tipo morto (ver item acima).
- `src/types/postgres.d.ts` e `src/types/html2pdf.d.ts` — ambos corretos e enxutos (declaration modules para libs sem tipos oficiais), sem `any` solto (usam `Record<string, unknown>` e generics). Nenhum achado.
- Consistência `interface` vs `type`: o padrão do projeto (interface para shapes de objeto/entidade, type para unions/aliases) é respeitado nos 4 arquivos sem exceções.

## Ponto 12 — Testes (src/test)

### Cobertura ausente em lógica crítica (fora de src/test, mas expõe o gap real)
- **`src/hooks/auth/useAuthRBAC.ts`** (214 linhas) — nenhum teste unitário. Concentra: derivação de `baseRole`, `canSimulate` (quem pode simular outro papel), `startSimulation`/`stopSimulation`, e o cálculo de `effectiveRole`/`effectiveProfile`/`effectiveStoreId` que decide qual identidade e loja o app usa em toda a sessão. É a peça mais sensível de autorização client-side do sistema e está sem cobertura alguma. O único teste relacionado a auth em `src/hooks/useAuth.test.ts` cobre apenas `pickSimulationStore` (função auxiliar de escolha de loja-sandbox), não a lógica de RBAC em si.
- **`src/hooks/auth/useAuthProfile.ts`, `useAuthSession.ts`, `useAuthActions.ts`** (200/124/203 linhas) — mesmo diagnóstico: zero testes. `useAuthActions` provavelmente cobre login/logout/troca de senha (fluxo de `auth-first-login.playwright.ts` testa o comportamento via UI/E2E, mas não há teste unitário isolado da lógica do hook).
- **`src/lib/services/checkin-service.ts`** — sem teste, apesar de ser lógica de negócio sensível (deduplicação de lançamento diário + log de auditoria + insert). Está marcado `@deprecated` no próprio arquivo (Story 1.2/1.3, caminho legado pré-RPC `submit_checkin`), mas segue importado por `AiDiagnostics.tsx` e `MorningReport.tsx`; enquanto não for removido, é código de produção sem rede de segurança.
- **`src/hooks/useCheckinAuditor.ts`** — hook de auditoria de check-in, sem teste correspondente.
- Módulos de feature com zero testes e volume relevante de arquivos: `dashboard-loja` (31 arquivos, inclui `useDashboardLojaData`, `useStoreActions`, `useStoreResolution` — hooks de resolução de loja/ações administrativas), `sales-performance` (23 arquivos), `consultoria` (17 arquivos), `configuracoes` (18 arquivos), `consultoria-cliente` (16 arquivos), `agenda-admin` (16 arquivos — mitigado parcialmente por `src/test/agenda-admin/useAgendaAdminForms.test.tsx`, mas cobre só 1 de 16 arquivos), `lojas` (12), `rotina-gerente` (12). Cálculo de score (`calcularScoreMX`), funil (`calcularFunil`, `gerarDiagnosticoMX`) e boa parte do CRM (`useMeuScore`, `useScoreRotina`, `cadencia*`) têm teste, então o "cálculo de score" citado no pedido está coberto — o gap real está em auth e nos módulos de dashboard/consultoria administrativa.

### Testes que só verificam "renderiza sem quebrar"
- `src/test/molecules/PageHeader.test.tsx:42-45` — teste `"does not render description when omitted"` faz `expect(screen.queryByText("Just Title")).toBeDefined()`; `queryByText` sempre retorna `Element | null`, e `null` também satisfaz `toBeDefined()`. O teste passa independente do comportamento real (não verifica ausência de descrição, que é o que o nome promete). Deveria ser `.toBeNull()` seguido de checagem de que a descrição específica não está presente.
- `src/test/atoms/EmptyState.test.tsx:21-28` — `"does not render description when omitted"` usa uma heurística frágil (`hasDescription || descs.length === 0`) que torna o assert quase sempre `true` independentemente do comportamento; não valida de fato a ausência da descrição.
- `src/test/e2e/smoke-flows.playwright.ts` combinado com `components.playwright.ts`, `consultoria.playwright.ts` e `agenda.playwright.ts` — múltiplos testes usam o padrão `expect(x).toBe(true).catch(() => { expect(true).toBe(true) })` ou `expect(hasEmptyState || hasVisitCard || true).toBe(true)` (ex.: `components.playwright.ts:59-75`, `:162-169`; `consultoria.playwright.ts:48-56`, `:83-101`, `:104-123`, `:125-148`, `:150-184`, `:186-207`, `:209-231`; `agenda.playwright.ts:120-130`, `:139-150`). Esse padrão faz o teste "passar" tanto se o elemento existir quanto se não existir — é teste que não pode falhar, apenas documenta um caminho feliz opcional. Ponto de qualidade mais recorrente da suíte E2E.

### Mocks que escondem bug real
- `src/test/molecules/ModalTrigger.test.tsx:7-27` e `src/test/organisms/Modal.test.tsx:7-27` — mockam `@radix-ui/react-dialog` inteiro com uma implementação simplificada própria (Root/Portal/Overlay/Content/Title/Description/Close reimplementados como `div`/`button` genéricos). Isso significa que os testes nunca exercitam o comportamento real do Radix (focus trap, portal para `document.body`, `aria-modal`, fechamento por `Escape`/clique fora). Um bug de acessibilidade ou de integração real com Radix não seria pego por esses testes — eles testam a lógica de `ModalTrigger`/`Modal` isoladamente, mas dão falsa confiança de que o modal "funciona" de ponta a ponta.
- Nos testes de segurança (`src/test/security/*.playwright.ts`) e E2E, não há mock do Supabase — usam client real contra banco de teste. Isso é positivo para não esconder bugs de RLS, mas nenhum teste unitário equivalente existe para simular RLS negado sem depender de rede (todos os testes de RLS são E2E, tornando a suíte lenta e dependente de ambiente/seed disponível).

### Testes frágeis
- `src/test/security/RLS-Isolation.playwright.ts:16-42` e `perfis-reais-e-dados-sensiveis.playwright.ts:46-62` — dependem de contas fixas hardcoded (`vendedor@mxgestaopreditiva.com.br`, `dono@...`, `gerente@...`) existentes no ambiente de teste, e de que `vinculos_loja` do vendedor já tenha ao menos 1 registro (`vinculos?.[0]?.store_id`). Se o seed mudar ou a conta for removida, o teste quebra sem indicar claramente que é problema de fixture, não de regressão.
- `src/test/organisms/Modal.test.tsx:116-128` — `"calls onClose when close button clicked"` localiza o botão de fechar via `closeButtons.find((b) => b.querySelector("svg"))`, ou seja, depende de detalhe de implementação (existência de um SVG dentro do botão) em vez de um `aria-label`/`data-testid` estável; qualquer botão adicional com ícone dentro do modal quebraria o teste por ambiguidade.
- `src/test/agenda.playwright.ts:63-75` (`"clicking date filter changes active state"`) e `:120-130` (navegação prev/next) usam `.catch(() => {})` engolindo falhas de clique/timeout e depois fazem assert sobre `typeof isActive === 'boolean'` — isso é sempre verdadeiro (booleano é sempre booleano), tornando o teste incapaz de detectar regressão de fato.
- `src/test/e2e-helpers/supabase-admin.ts` é usado por praticamente toda a suíte E2E/security para criar/deletar usuários reais via `service_role`; qualquer teste que falhe antes do `test.afterEach` de limpeza (ex.: exceção síncrona) deixa usuários órfãos no banco de teste — não há `try/finally` de rede de segurança em vários specs (ex.: `agenda-filters.playwright.ts` limpa em `afterEach`, mas `mx-consultoria-role-smoke.playwright.ts` só deleta `adminMaster`, não lida com falha parcial de criação).
