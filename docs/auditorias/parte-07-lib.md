# Parte 7 — src/lib (agenda, consultoria, automation, raiz)

## Resumo

Escopo auditado (~8.800 linhas): `src/lib/agenda/`, `src/lib/consultoria/`, `src/lib/automation/` e os arquivos soltos na raiz de `src/lib/*.ts` + seus `.test.ts`. O achado mais grave é que **todo o diretório `src/lib/automation/` (13 arquivos, ~600 linhas) é código morto** — nenhum ponto de entrada do app (páginas, hooks, `api/`, `vercel.json`) o importa ou aciona; ele depende de APIs Node (`node-cron`, `process.env`) incompatíveis com o runtime Vite/browser do resto do projeto. `calculations.ts` está limpo, tipado e bem testado; `central-mx-engine.ts` (o motor de score/alertas mais crítico do domínio) tem apenas 3 blocos de teste que verificam forma/estrutura, não a lógica de pontuação e alertas em si. Não há uso de `any` em todo o escopo. Há duplicação recorrente de uma função de normalização de texto (remover acentos) em 4 arquivos diferentes.

## 1. DRY
- `src/lib/automation/cron-scheduler.ts:12-21` e `src/lib/automation/weekly/feedback-engine.ts:10-19` — `getRelationUserName` e `firstRelation` são definidas de forma idêntica nos dois arquivos. Extrair para um helper compartilhado (ex. `src/lib/automation/relations.ts`), se o diretório for mantido.
- `src/lib/automation/email/templates/matinal.ts:5-12` e `src/lib/automation/email/templates/weekly-feedback.ts:1-8` — `escapeHtml` duplicada byte a byte. Mover para um util comum de templates de e-mail.
- Normalização de texto (remover acentos/minúsculas) reimplementada de forma independente em 4 lugares: `src/lib/consultoria/pmr-engine.ts:140-145` (`normalizeText`), `src/lib/development-content.ts:136-141` (`normalize`), `src/lib/pdi-evolution.ts:169-176` (`normalizarCompetencia`) e `src/lib/utils.ts:65-76` (dentro de `slugify`). Todas usam o mesmo padrão `.normalize('NFD').replace(/[̀-ͯ]/g, '')...`. Sugestão: extrair `stripAccents(value: string): string` para `src/lib/utils.ts` e reusar nos 4 pontos.
- `src/lib/agenda/admin-master.ts:1-8` (`DEFAULT_ADMIN_MASTER_EMAILS`) duplica a mesma lista hardcoded de e-mails presente em `supabase/functions/_shared/google_calendar_privacy.ts:1-7`. Frontend e edge function mantêm cópias independentes da allowlist — risco de drift ao adicionar/remover um admin master (precisa lembrar de editar os dois lados). Não há como compartilhar módulo entre Vite e Deno diretamente, mas vale documentar a dependência ou gerar uma a partir da outra em build/CI.

## 2. Dead code
- **`src/lib/automation/` inteiro está morto.** Confirmado por grep: nenhum arquivo fora de `src/lib/automation/` importa `startAutomationJobs`, `runMatinalWorkflow`, `runWeeklyFeedbackWorkflow`, `runMonthlyCloseWorkflow`, `getAdminExecutiveOverview`, `trackTrainingConsumption`, `getTrainingByBottleneck`, `generateWhatsAppScript`, `sendEmailReport` ou `generateFeedbackXlsx`. Não existe rota `api/` nem `crons` em `vercel.json` acionando o módulo. Os arquivos usam `node-cron` e `process.env.RESEND_API_KEY` (APIs de Node, não de browser/Vite), o que sugere que foi desenhado para rodar como job separado que nunca foi ligado. Afeta: `src/lib/automation/index.ts`, `scheduler.ts`, `cron-scheduler.ts`, `logger.ts`, `admin/overview.ts`, `email/sender.ts`, `email/templates/matinal.ts`, `email/templates/weekly-feedback.ts`, `monthly/close-engine.ts`, `reports/xlsx-generator.ts` (incl. `generateFeedbackXlsx`, nunca chamada mesmo dentro do próprio diretório), `training/engine.ts`, `weekly/feedback-engine.ts`, `whatsapp/script-generator.ts`. Consequência colateral: as dependências `node-cron` e `exceljs` em `package.json` só são usadas por este código morto (`exceljs` não aparece em nenhum outro arquivo de `src`).
- `src/lib/morning-report.test.ts` não tem um `morning-report.ts` correspondente — na verdade importa e testa `formatWhatsAppMorningReport` de `./calculations`. Não é dead code (é teste válido), mas o nome do arquivo é enganoso; renomear para algo como `calculations.morning-report.test.ts` ou mover os testes para `calculations.test.ts` evitaria a falsa impressão de arquivo-fonte ausente.
- `src/lib/agenda/google-calendar-privacy.test.ts`, `google-calendar-sync-rules.test.ts`, `google-meet-cohost-rules.test.ts` — mesma situação: testam código de `supabase/functions/_shared/*`, não de `src/lib/agenda`. Não são dead code, mas a localização é inconsistente com o restante do diretório `agenda/` (só tem `admin-master.ts` como fonte real). Ver também item 10.

## 3. TypeScript
- Nenhum uso de `any` (`: any`, `as any`, `any[]`) encontrado em todo o escopo (grep confirmado). Positivo, principalmente em `calculations.ts` e `central-mx-engine.ts`, que são totalmente tipados com tipos explícitos de retorno.
- `src/lib/consultoria/pmr-engine.ts:134-138` (`numberValue`) aceita `unknown` e faz coerção runtime — correto para dados de fonte externa, mas o retorno poderia ser documentado com um tipo literal `number` já é; sem problema real.
- Nenhuma função pública em `calculations.ts` ou `central-mx-engine.ts` está sem tipo de retorno explícito nas assinaturas mais relevantes; funções internas menores (ex. `clampScore`, `average` em `central-mx-engine.ts:225-233`) omitem o tipo de retorno mas o TS infere corretamente `number` — estilo aceitável, não é um risco real de tipagem.

## 6. Hooks
N/A — nenhum hook React customizado neste escopo (`src/lib/agenda`, `consultoria`, `automation`, raiz de `lib`). Os hooks que consomem essas funções (`useCentralMxAlerts`, etc.) ficam em `src/features/*` e `src/hooks/*`, fora do escopo desta auditoria.

## 8. Erros
- `src/lib/automation/cron-scheduler.ts:31-35,63-69` — `const { data: lojas } = await supabase.from('lojas')...` e `const { data: members } = await supabase.from('vinculos_loja')...` descartam `error` do retorno do Supabase. Se a query falhar, `lojas`/`members` fica `null`/`undefined` e a função apenas retorna silenciosamente (`if (!lojas) return`), sem log do motivo. Comparar com `src/lib/automation/weekly/feedback-engine.ts:34-41`, que faz o certo (`const { data: lojas, error: storeErr } = ...` + `console.error`). Mesmo sendo código morto (item 2), a inconsistência é um padrão a evitar se o diretório for reativado ou usado como referência.
- `src/lib/automation/monthly/close-engine.ts:23-31` trata erro corretamente (`storesError` logado), mas o `catch` por loja (`:67-69`) apenas empilha `status: 'error'` no array de resultado sem logar — falha silenciosa para quem não inspecionar o retorno.
- `src/lib/consultoria/evidence-attachments.ts:38-47` (`downloadEvidenceAttachment`) chama `createEvidenceSignedUrl` sem `try/catch` local; se falhar, propaga para o chamador (aceitável, mas assimétrico em relação a `openEvidenceAttachment`, que trata o erro localmente para fechar a aba aberta antes de repropagar). Não é bug, apenas falta de padronização.
- Restante do escopo (`calculations.ts`, `central-mx-engine.ts`, `mx-executive-foundation.ts`, `daily-routine.ts`, `pdi-evolution.ts`, `storeSalesRules.ts`, consultoria/*) é composto de funções puras sem I/O — não há chamadas Supabase para auditar ali.

## 9. Performance
- `buildCentralMxEngine` (`src/lib/central-mx-engine.ts:431`) é uma função pura relativamente pesada (percorre 45 indicadores, 6 departamentos, monta scores e alertas a cada chamada) mas os dois pontos de consumo (`OwnerExecutiveCockpit.tsx`, `useCentralMxAlerts.ts`) já envolvem a chamada em `useMemo` com dependências corretas — sem re-cálculo desnecessário a cada render. Nenhum problema de performance identificado nos arquivos de cálculo/engine.
- `src/lib/consultoria/pmr-engine.ts` — `derivePmrMetricResults` faz múltiplos `.filter()`/`.reduce()` sequenciais sobre os mesmos arrays (`salesRows` percorrido ~7 vezes para classificar canal: `internetSalesFromEntries`, `doorSales`, `referralSales`, `companyWalletSales`, `sellerWalletSales`). Para volumes de vendas mensais por loja (dezenas/centenas de linhas) isso é irrelevante na prática, mas poderia ser um único loop de classificação se o volume crescer.
- Nenhum loop O(n²) ou recomputação cara identificada nos arquivos de rotina/PDI (`daily-routine.ts`, `pdi-evolution.ts`) — operam sobre arrays pequenos (sessões de PDI, slots de rotina).

## 10. Organização
- `src/lib/automation/` deveria ser removido (ver item 2) ou, se houver intenção real de reativá-lo como job server-side, movido para fora de `src/lib` (ex. um pacote/worker separado), já que hoje ele quebra a fronteira "tudo em `src/lib` roda no browser" ao depender de `node-cron`/`process.env`.
- `src/lib/consultoria/pmr-engine.ts` (842 linhas) concentra três responsabilidades bem distintas: (1) derivação de métricas a partir de dados brutos (`derivePmrMetricResults`, `computeDreLike`), (2) tabelas estáticas de conteúdo/recomendação (`PMR_IMPLEMENTATION_SCHEDULE`, `PMR_RISK_CONTINGENCIES`, `METRIC_ACTIONS`, `DIAGNOSTIC_ACTIONS` — só essas quatro constantes somam ~250 linhas) e (3) geração de markdown/plano estratégico (`buildMarkdown`, `buildPmrStrategicPlan`). Separar as constantes de conteúdo em `pmr-engine.content.ts` (ou `pmr-action-catalog.ts`) reduziria o arquivo principal para ~600 linhas e isolaria o "dado" da "lógica", facilitando manutenção do catálogo de ações sem tocar no motor de cálculo.
- Inconsistência de localização: `src/lib/agenda/google-calendar-privacy.test.ts`, `google-calendar-sync-rules.test.ts` e `google-meet-cohost-rules.test.ts` testam módulos de `supabase/functions/_shared/`, não código de `src/lib/agenda`. Isso deixa `src/lib/agenda/` com 4 arquivos de teste para apenas 1 arquivo de fonte real (`admin-master.ts`). Mover esses três testes para perto do código que testam (`supabase/functions/_shared/*.test.ts` ou um diretório `supabase/functions/_shared/__tests__/`) tornaria a árvore mais previsível — hoje quem abre `src/lib/agenda/` esperando ver 4 fontes correspondentes se depara com 3 "órfãos".
- A mistura de muitos arquivos soltos na raiz de `src/lib/*.ts` (calculations, central-mx-engine, daily-routine, pdi-evolution, pdi-self-assessment, storeSalesRules, mx-executive-foundation, development-content, team-contacts-export, avatar, utils, export, format, feature-flags, query-client, migration-validator, supabase — 17 arquivos de domínio + ~19 `*-migration.test.ts`) versus pastas organizadas (`agenda/`, `consultoria/`, `automation/`) é inconsistente. Vários desses arquivos soltos são claramente do domínio "MX Score/Engine" (`calculations.ts`, `central-mx-engine.ts`, `mx-executive-foundation.ts`, `storeSalesRules.ts`) e poderiam viver em uma pasta `src/lib/mx-engine/` ou `src/lib/score/`, espelhando o padrão já usado por `agenda/`/`consultoria/`/`automation/`.

## 12. Testes
- `src/lib/calculations.test.ts` (251 linhas / 20 blocos `test`) cobre praticamente todas as funções exportadas de `calculations.ts` com múltiplos cenários e casos de borda (meta zero, dias restantes zero, arredondamento, funil vazio, venda loja). Boa cobertura, não é smoke test.
- `src/lib/central-mx-engine.test.ts` (110 linhas / apenas 3 blocos `test`) é claramente **subdimensionado para a criticidade do arquivo** (514 linhas, motor de score/alertas/plano de ação usado no dashboard executivo). Os testes verificam: (1) que o catálogo tem 45 indicadores únicos distribuídos nos 6 departamentos, (2) que `buildCentralMxEngine` retorna a forma esperada (`alerts.length > 0`, contagens de departamentos/scores/ações) sem checar o **conteúdo** de nenhum alerta específico, e (3) que o catálogo está alinhado com uma migration SQL. Nenhum teste cobre: `scoreFromActual` (a fórmula de ratio meta/realizado com `targetDirection` invertido para métricas "lower is better"), os gatilhos específicos de `buildAlerts` (ex.: garantir que `leadRate < benchmark` realmente gera o alerta `critical` de `lead_to_schedule_rate` e não outro), o cálculo de `individualScores` (pesos de `attainment`/`checkedIn`/`goal`), nem o fallback de "alerta positivo" quando `alerts.length === 0`. Recomenda-se adicionar testes unitários para essas funções internas (hoje não exportadas — precisariam ser exportadas para teste direto, ou testadas indiretamente via `buildCentralMxEngine` com inputs desenhados para cada branch).
- `src/lib/consultoria/pmr-engine.test.ts` (288 linhas / 7 testes) para um arquivo de 842 linhas com lógica bem mais ramificada (`derivePmrMetricResults`, `buildPmrStrategicPlan`, `recommendationsFromMetrics`, `recommendationsFromDiagnostics`) — cobertura razoável mas também mais rasa que o tamanho do arquivo sugere; não bloqueante pois não foi apontado como crítico pelo usuário, mas vale nota.
- `src/lib/pdi-evolution.test.ts` (2 testes) e `src/lib/storeSalesRules.test.ts` (2 testes) são enxutos; para `storeSalesRules.ts` (23 linhas, função única de merge de defaults) é proporcional. Para `pdi-evolution.ts` (181 linhas, lógica de status evoluindo/estagnado/queda e ordenação de highlights) 2 testes é pouco — não cobre, por exemplo, o critério de desempate por `Math.abs(delta)` em `buildPDIEvolution` (`central-mx-engine.ts` não, é em `pdi-evolution.ts:96-102`) nem o caso `comparavel: false` com menos de 2 sessões.
- Os arquivos `*-migration.test.ts` (19 arquivos na raiz) são testes de migração de dados legados, conforme já esperado — não são dead code mesmo sem "consumidor" fora de si mesmos.
